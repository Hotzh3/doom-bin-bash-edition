# doom-bin-bash-edition

[![CI](https://github.com/Hotzh3/doom-bin-bash-edition/actions/workflows/ci.yml/badge.svg)](https://github.com/Hotzh3/doom-bin-bash-edition/actions/workflows/ci.yml)
[![Release](https://github.com/Hotzh3/doom-bin-bash-edition/actions/workflows/release.yml/badge.svg)](https://github.com/Hotzh3/doom-bin-bash-edition/actions/workflows/release.yml)
[![Latest Release](https://img.shields.io/github/v/release/Hotzh3/doom-bin-bash-edition)](https://github.com/Hotzh3/doom-bin-bash-edition/releases)

**Retro-horror raycast FPS** — a **browser-playable** vertical slice built with **Phaser 3**, **TypeScript**, and **Vite**. **`RaycastScene`** is the pitch: Episode 1 (five sectors + boss), optional **World 2** abyss arc (**four** data-driven sectors), **`GameDirector`** pacing, HUD / minimap, sector reports with score and rank, and generated WebAudio. **`ArenaScene`** is a **secondary** 2D sandbox for regression coverage.

**Status:** Portfolio-ready — core logic covered by **Vitest**, ESLint, production **Vite** build, GitHub Actions CI.

**Media:** Gameplay captures and moodboard references live in `docs/assets/` and the phase docs; the README now stays text-first.
Capture workflow: [`docs/media/capture-guide.md`](docs/media/capture-guide.md).

---

## Disclaimer

Portfolio / learning project. Inspired by the **feel** of classic retro FPS titles; **not** affiliated with Doom or Doom 64. No reuse of their code, assets, maps, names, sprites, sounds, or copyrighted content.

## Clean-room boundary

No copied gameplay assets, maps, proprietary data, or reverse-engineered implementations. No content from DOOM64-RE. References are **high-level only** (movement clarity, strafe play, pacing, readable horror atmosphere). Layouts, tuning, names, and visuals in this repo are original.

---

## What ships today

- **Raycast episode:** menu → terminal prologue → **Episode 1** (five sectors + **Volt Archon** boss) → optional **World 2** four-sector rift (data-driven).
- **Gameplay:** WASD, mouse / keys look, hitscan weapons, doors / keys / secrets, ambush triggers, objectives, pause / death / clear flows.
- **Presentation:** Compact HUD, minimap, difficulty presets, challenge modifiers, sector report (score, high score, rank, par, combo, medals).
- **Systems:** `GameDirector` pacing (calm → pressure → ambush → recovery), enemy roles (`GRUNT`, `STALKER`, `RANGED`, `BRUTE`).
- **Quality:** Broad **unit tests** for raycast, combat, director, and presentation logic without requiring a live canvas.
- **Arena:** Local 2-player sandbox — preserved, not the primary pitch.

## Quick start

```bash
npm ci
npm run dev
```

Open the Vite URL (usually `http://localhost:5173`). From the menu: **`A`** or **“Press A: 3D Mode”** starts the raycast path (prologue → episode). **`B`** opens the 2D arena. **`D`** cycles raycast difficulty.

## Runtime strategy

- `npm run build` compiles the browser client and writes `dist/runtime-manifest.json`.
- `npm run build:preview` builds with a GitHub Pages style base path for static deploy checks.
- `npm run build:release` labels the artifact as a release build.
- `npm run preview:dist` serves the built `dist/` folder locally.
- Runtime metadata is surfaced in the menu footer so the current build is visible in-browser.

## CI/CD (GitHub Actions)

- **CI** workflow: [`.github/workflows/ci.yml`](.github/workflows/ci.yml)
  - Runs on push and pull request.
  - Executes `npm ci`, `npm run test`, `npm run lint`, `npm run build`, and `npm audit --audit-level=high`.
- **Release** workflow: [`.github/workflows/release.yml`](.github/workflows/release.yml)
  - Runs on pushes to `main`, tags `v*`, and manual dispatch.
  - Uploads downloadable `dist/` and `.zip` artifacts to GitHub Actions.
  - Creates GitHub Releases automatically for version tags with autogenerated notes.
- **CD** workflow: [`.github/workflows/cd-pages.yml`](.github/workflows/cd-pages.yml)
  - Runs on push to `main` and manual `workflow_dispatch`.
  - Executes `npm ci`, `npm run test`, `npm run lint`, `npm run build`.
  - Publishes `dist/` to GitHub Pages and uploads a `dist` artifact for traceability.

## Release workflow

`main` is the preview lane:

- every push uploads downloadable build artifacts in GitHub Actions
- GitHub Pages publishes the latest browser-playable version

Version tags are the release lane:

- create a tag like `v0.1.0`
- push it to GitHub
- Actions builds the game, uploads artifacts, creates a GitHub Release, and attaches a zip asset

Quick example:

```bash
git tag v0.1.0
git push origin v0.1.0
```

Release flow guide:
[docs/runtime/release-flow.md](docs/runtime/release-flow.md)

### GitHub Pages activation (one-time)

1. In GitHub: `Settings` → `Pages`.
2. Set **Source** to **GitHub Actions**.
3. Push/merge to `main` (or trigger the CD workflow manually).
4. Demo URL pattern: `https://<org-or-user>.github.io/<repo-name>/`.

`BASE_PATH` is set automatically in CD as `/${repo-name}/` for Vite builds deployed to Pages.
`dist/runtime-manifest.json` captures build metadata for each artifact.

## Technical highlights

- **Raycast column renderer** and authored maps / level data (no external map packs).
- **Separation** of presentation (`RaycastPresentation`, `RaycastRunSummary`) from simulation.
- **`GameDirector`** and pacing helpers covered by unit tests where logic is pure.
- **World 2** data in `RaycastWorldTwoLevels.ts` (re-exported from the level module).
- **No backend** — high score is **local** (`localStorage`).

## AI status

- **Applied gameplay AI (current):** `GameDirector`, enemy behavior decisions, dynamic pressure/recovery pacing, and boss phase behavior.
- **OpenAI / external AI API:** pending team decision. Not required to run or play the game.
- **Security rule:** do not commit API keys; if future optional setup is approved, use `.env.example` placeholders only.
