# Infrastructure (Current State)

This document formalizes the project infrastructure for academic delivery. It focuses on what is actually implemented today.

## Scope

- Product runtime is a browser game client (Phaser + TypeScript + Vite).
- There is no required backend service in the current playable scope.
- Infrastructure covers local development, Docker runtime, CI, and CD.

## Local Architecture

### Runtime model

- Game runs as static web assets + browser runtime logic.
- Main dev runtime uses Vite dev server.
- Production runtime uses built `dist/` static files.

### Toolchain

- Node.js 20.x
- npm (`npm ci` preferred for deterministic installs)
- Vite for dev/build/preview
- Vitest for tests
- ESLint for lint checks

## Ports

- Local dev (`npm run dev`): `5173` (default Vite port)
- Docker dev (`docker compose up`): `5173` exposed as `5173:5173`
- Local preview (`npm run docker:preview`): `5173`
- GitHub Pages deploy: standard HTTPS `443` on Pages domain

## Docker Runtime

### Files

- `Dockerfile`
- `docker-compose.yml`
- `.dockerignore`

### Behavior

- Base image: `node:20-alpine`
- Installs dependencies with `npm ci`
- Runs `npm run docker:dev` (`vite --host 0.0.0.0 --port 5173`)
- Exposes port `5173`

### Compose model

- Single service: `game`
- Bind mount project folder to `/app`
- Uses separate `/app/node_modules` volume to avoid host-container conflicts

## CI Pipeline

Workflow: `.github/workflows/ci.yml`

On `pull_request`, `push` to `main`, and manual dispatch:

1. Checkout
2. Setup Node 20
3. `npm ci`
4. `npm run test`
5. `npm run lint`
6. `npm run build`
7. `npm audit --audit-level=high`
8. Upload `ci-dist-<sha>` artifact on `main` pushes

Purpose: quality gate before merges/releases.

## CD / Deploy Pipeline

Workflow: `.github/workflows/cd-pages.yml`

On `push` to `main` and `workflow_dispatch`:

1. Checkout
2. Setup Node 20
3. Configure GitHub Pages
4. `npm ci`
5. `npm run test`
6. `npm run lint`
7. `npm run build` with `BASE_PATH=/<repo-name>/`
8. Upload `dist/` as Pages artifact
9. Upload `dist/` as build artifact (`upload-artifact`)
10. Deploy to GitHub Pages

Purpose: simple, auditable static deployment for demos.

## Workflow matrix (quick view)

| Workflow | Trigger | Main output |
|---|---|---|
| `ci.yml` | PRs, push `main`, manual | quality gate (`test/lint/build/audit`) + `ci-dist` artifact on `main` |
| `cd-pages.yml` | push `main`, manual | GitHub Pages deployment + `dist-<sha>` artifact |
| `release.yml` | push `main`, tags `v*`, manual | release `dist` + zip artifacts; GitHub Release on tags |

## Release Pipeline

Workflow: `.github/workflows/release.yml`

On `push` to `main`, version tags `v*`, and `workflow_dispatch`:

1. Checkout
2. Setup Node 20
3. `npm ci`
4. `npm run test`
5. `npm run lint`
6. `npm run build`
7. Package `dist/` into a versioned `.zip`
8. Upload `dist/` and `.zip` as GitHub Actions artifacts
9. If the trigger is a tag, create a GitHub Release with autogenerated notes and attach the zip asset

Purpose: keep every mainline build downloadable, and every tagged build formally releasable.

## Services Not Used (By Design)

Not used in current scope:

- Dedicated backend API
- Database (SQL/NoSQL)
- Auth service
- Matchmaking service
- Online leaderboard service
- Cloud game server runtime

Why:

- Current product is a local/browser-first vertical slice.
- Score persistence is local (`localStorage`), not networked.
- Avoids overengineering for current academic requirements.

## Demo Runbook

### Local (recommended in classroom)

```bash
npm ci
npm run dev
```

Open `http://localhost:5173`.

### Docker

```bash
docker compose up --build
```

Open `http://127.0.0.1:5173`.

Validate runtime response:

```bash
curl -I http://127.0.0.1:5173
```

Expected key line:

- `HTTP/1.1 200 OK`

Stop:

```bash
docker compose down
```

### Validation commands

```bash
npm run test
npm run lint
npm run build
```

## Text Diagram

### Current

```text
Developer -> GitHub -> CI (test/lint/build/audit) -> Build artifact / GitHub Pages
Player -> Browser -> Vite dev server OR static dist build
```

### Optional future (not implemented yet)

```text
Host client -> WebSocket server -> remote clients
```

## Extension Path to Multiplayer Host/Client

If multiplayer is added later, minimal infra evolution could be:

1. Add lightweight authoritative host process (Node + WebSocket).
2. Keep game client static (same Vite/build flow).
3. Add optional state relay service for remote peers.
4. Add separate CI jobs for server tests and protocol contracts.
5. Keep CD split: static client deploy + optional server deploy.

This is intentionally future-facing and not part of current runtime.
